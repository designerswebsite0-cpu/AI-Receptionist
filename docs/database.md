# database.md
# AI Receptionist Platform Database Specification (v1)

> Status: Living Database Specification
> Database: PostgreSQL (Supabase)
> Architecture: Single-resort deployment template (Phase 2.5) — one database per resort, not shared multi-tenant SaaS
> Shared By: Chat + Calls + Customer360 + Knowledge Intelligence Engine (KIE)

---

# 1. Database Philosophy

The database is the single source of truth for the platform.

Design Principles:

- Single-resort per deployment (no tenant_id — see §3)
- UUID primary keys
- Row Level Security (RLS)
- Highly normalized
- Audit friendly
- AI friendly
- Future proof
- Soft deletes where appropriate

# 3. Resort Configuration Module

**Status: implemented (Phase 2.5).** Replaces the earlier multi-tenant
`tenants`/`tenant_settings`/`tenant_members`/`tenant_roles`/`tenant_permissions`
tables, all removed in migration `0008`. This deployment serves exactly one
resort, so there is no tenant_id anywhere in the schema and no per-row
ownership to enforce — see product_decisions.md for the full rationale.

Tables (implemented)

- resort_settings — exactly one row per deployment, enforced by a
  `singleton` boolean under a UNIQUE constraint (not just an application
  check): resort_name, legal_name, description, address/city/state/
  country/postal_code, phone/email/whatsapp, timezone, currency,
  default_language, check_in_time/check_out_time, logo_url,
  primary_brand_color/secondary_brand_color, website_url, settings_metadata
  (JSONB)

Access model: every authenticated application user has full access (no
roles/permissions table — see rules.md §4). `users` (below) still exists as
the profile mirror of Supabase `auth.users`, just without any
membership/role join table pointing at it.

# 4. Customer360 Module

**Status: foundation implemented in Phase 2** (structure only — no AI
summaries/inferences yet; those need source/confidence/verification
tracking per rules.md §6 and land with the AI Orchestration Engine).

Tables (implemented)

- customers — identity, preferred_language/channel, lifetime_value,
  loyalty_reference, and two JSONB buckets (`preferences`,
  `resort_preferences`) rather than many single-purpose columns, since the
  shape of that data isn't settled until the AI memory layer exists
- customer_contacts — unified phone/email/WhatsApp identity table, unique
  per (contact_type, value) — globally unique within this deployment's one
  resort; this is how "one customer profile across channels"
  (architecture.md §4.2) actually resolves
- customer_notes — staff-authored, attributed (author_user_id); a verified
  fact, never an AI inference
- customer_tags

Deliberately not separate tables yet — derived instead of duplicated:

- "Previous stays" — will be a query against `bookings` (Phase 7, not yet
  built) filtered by customer_id, not a column here
- "Communication history" — a query against `conversations`/`messages`
  (below) filtered by customer_id

Originally planned but not yet needed / superseded by the above:
customer_preferences, customer_behaviour, customer_addresses,
customer_scores, customer_favourite_products, customer_favourite_services,
customer_booking_history, customer_order_history, customer_ai_summary,
customer_timelines — most of this is either folded into the `preferences`/
`resort_preferences` JSONB for now, or deferred to the phase that actually
populates it (scores/AI summary → Phase 4; booking/order history → Phase 7).

Shared across Chat and Calls.

# 5. Conversation Module

**Status: foundation implemented in Phase 2.**

Tables (implemented)

- conversations — channel, status (lifecycle/queue state), current_state
  (dialogue state — independent of status, see architecture.md §4.4),
  assigned_agent_id, priority, ai_active/human_active flags, tags (JSONB),
  conversation_metadata (JSONB)
- conversation_state_events — audit trail of every current_state
  transition (from_state, to_state, changed_by, metadata)
- messages — channel-independent; direction, sender_type, content_type,
  delivery_status, read_at, external_message_id (idempotency, ready for
  Phase 6 WhatsApp webhooks)
- message_attachments — storage_path only (private Supabase Storage;
  signed URLs generated by backend code, never a public URL)

Originally planned, not yet needed: conversation_participants (multi-party
conversations aren't a current requirement), conversation_labels (superseded
by `conversations.tags` JSONB), conversation_assignments (superseded by
`conversations.assigned_agent_id`), conversation_events (superseded by
audit_logs + conversation_state_events), conversation_ai_context (arrives
with the AI Orchestration Engine, Phase 4).

Supports:

- WhatsApp
- Web Chat
- Voice (schema is channel-neutral; no voice-specific columns needed)
