# Phase 5 — Website Codebase Audit

Date: 2026-07-18
Source: `RKPR-WEBSITE-main.zip` (uploaded, no `.git` history — a GitHub branch
export). Extracted and reviewed in full, then imported into this monorepo at
[apps/website](../../apps/website) as a new npm workspace (`apps/website`,
`web_app@0.1.0`). This is a **new location decision**, recorded here since
Step 1 explicitly asks for it: the zip was not part of any existing repo
structure, and Phase 5 needs it to share this platform's env-var conventions,
CI, and deployment story. `apps/widget` (an existing empty placeholder from
the Phase 1 scaffold) was left untouched — it was reserved for a future
embeddable third-party widget, a different concept from this resort's own
full marketing site.

Excluded on import: `scratch/` (629 KB of one-off content-extraction Python
scripts and duplicated raw RAG text, not referenced anywhere by `src/`),
`Website Sources Register.xlsx`, and `package-lock.json` (regenerated by the
workspace root install). `public/Images/` (61 MB of real resort photography)
was kept in full — it's genuine site content.

Baseline verified before any code changes: `npm install` at the repo root
succeeds (61 packages added), `npm run website:build` succeeds (48 static/SSG
routes + 2 dynamic API routes), confirming the uploaded code is not currently
broken. `npm run website:lint` fails: **33 pre-existing errors, 57 warnings**
(unescaped apostrophes in JSX text, unused lucide-react icon imports) spread
across pages Phase 5 does not touch. These predate this audit and are not
caused by anything in Phase 5. See "Proposed implementation plan" below for
how this is handled.

## 2. The existing chat widget — critical finding

[apps/website/src/components/ChatWidget.tsx](../../apps/website/src/components/ChatWidget.tsx)
is a **fully client-rendered, already-polished UI** (branded as "Aranya, AI
Concierge Receptionist", forest/bronze themed, floating launcher bottom-right,
framer-motion open/close, unread-ping badge, typing indicator, auto-scroll).
Structurally this is a good starting point for Phase 5's UI layer — the
visual language, layout, and animation choices already match the brand and
can mostly be reused.

However, it POSTs to [apps/website/src/app/api/chat/route.ts](../../apps/website/src/app/api/chat/route.ts),
which is **not a real backend integration at all** — it's ~170 lines of
`if (query.includes("..."))` keyword matching that fabricates:

- Specific room rates ("INR 9,500/night"), spa prices, laundry prices, transfer fares — invented, unsourced numbers.
- Fake "emergency transfer" / "human handoff" / "duty manager escalation" messages that don't touch any real conversation, customer, or handoff state — they're just canned strings.
- A hard-coded default fallback reply.

This directly violates this project's `docs/CLAUDE.md` rule ("The LLM must
never invent bookings/prices/policies/availability") and every explicit
instruction in the Phase 5 brief to route through the real Phase 4
orchestration pipeline. **This route must be replaced, not extended** — its
business logic is not reusable in any form (it doesn't even call an LLM).
`ChatWidget.tsx`'s JSX/styling is reusable; its data layer is not.

`/api/enquiry` ([apps/website/src/app/api/enquiry/route.ts](../../apps/website/src/app/api/enquiry/route.ts))
is a separate, already zod-validated contact form endpoint. It's also fully
mocked (returns a random fake reference number, persists nothing) but it is
**out of scope for Phase 5** (it's a general contact form, not the chat
channel) — flagged here for the record, not modified.

## 3. Backend integration points (already built, Phase 3/4)

Every existing FastAPI endpoint (`auth`, `resort`, `customers`,
`conversations`, `knowledge`, `orchestration`) requires
`Depends(get_current_user)` — a verified Supabase JWT
([apps/api/app/deps.py](../../apps/api/app/deps.py)). There is **no anonymous
access path anywhere in the backend today**. This confirms the Phase 5 brief's
premise: the browser cannot call these staff-only endpoints directly, and a
new public boundary is required.

Relevant existing pieces to build on, not duplicate:

- `app.orchestration.pipeline.orchestrate()` — the single channel-neutral entry point. Already accepts `channel` (webchat is already a valid value — see below) and is idempotent on `message_id`.
- `app.orchestration.router` (`/api/v1/orchestration/*`) — the staff/dashboard-facing wrapper over `orchestrate()`. Its request shape (`ProcessMessageRequest{message_id, channel}`) assumes the message is **already persisted** before orchestration runs; the public webchat layer must persist the guest's message the same way before invoking the pipeline, not pass raw text into a duplicate code path.
- `app.conversations.constants.CHANNELS = ("whatsapp", "webchat")` — `"webchat"` is **already** the canonical channel value; Phase 5 should use it as-is (the brief's suggested `website_chat` is not necessary — reusing the existing enum value avoids a migration).
- `app.customers.service.create_customer` / `app.customers.models.Customer` + `CustomerContact` — Customer 360 foundation, already supports contactless (no phone/email) customers and later contact addition via `add_contact`. No changes needed to accept an anonymous guest as a `Customer` row with zero contacts.
- `app.conversations.service.create_conversation` — accepts `customer_id` + `channel` + `priority`; conversations start `ai_active=True`, `current_state="greeting"`.
- `app.orchestration.schemas.ProcessMessageResponse` / `TurnCitationOut` — already shaped for citation display (`source_title`, `source_priority`, `authoritative`, `score` — no internal chunk/vector IDs leak through this schema today, which is the right shape to expose to guests too).
- `app.config.Settings` — `cors_allowed_origins` is already a configurable list; Redis is already optional-with-fallback (`redis_url: str | None`); `openai_api_key` / `groq_api_key` presence-only health check pattern already exists (`/orchestration/health/providers`).

### Gaps that must be built (nothing here exists yet)

- Any concept of an **anonymous** identity. `get_current_user` always requires a real Supabase session — a website guest has none.
- Any endpoint under a public/webchat prefix — router does not exist.
- Rate limiting of any kind beyond the auth-login-specific in-memory limiter mentioned in `config.py` (`auth_login_rate_limit_per_minute`) — there is no general-purpose rate-limit dependency, no Redis-backed limiter, nothing applied to any existing router.
- Streaming — `orchestrate()` returns one complete `OrchestrationResult` synchronously; there is no token-level or chunked streaming anywhere in the LLM provider layer (`app.orchestration.llm.*`). Implementing real token streaming would mean threading a callback/generator through `orchestrate()`, the tool-call round-trip, and guardrail validation — a nontrivial change to a pipeline that was just hardened and real-data-validated in Phase 4. Per the brief's own instruction ("If the current orchestration endpoint does not safely support streaming, do not implement a fragile fake streaming system... use a clean loading state"), Phase 5 will use a single request/response call with a typing indicator, and document real streaming as a deferred enhancement.
- CSRF: cookies will be `SameSite=Lax` + `HttpOnly` for the session token; since the webchat API only accepts JSON POSTs (not form submissions) and is same-site-or-explicit-origin (CORS), CSRF risk is low but will be explicitly reasoned through in `WEBCHAT_SECURITY.md`.

## 5. Risks

1. **Fabricated content precedent** — the existing fake widget already trained a reviewer's eye on specific invented numbers (e.g. "INR 9,500/night"). The real widget must not accidentally reproduce these as if they were verified — it will only ever surface what the real RAG pipeline retrieves and cites.
2. **Anonymous IDOR** — a session token that's guessable or that leaks another guest's `conversation_id` would let one guest read another's chat. Mitigated by a high-entropy opaque token resolved server-side to a conversation, never trusting a client-supplied `conversation_id`/`customer_id` directly (detailed in `WEBCHAT_SECURITY.md`).
3. **Cost/abuse** — an anonymous, unauthenticated endpoint that triggers real OpenAI calls is a direct cost-abuse surface if unprotected. Rate limiting is not optional here (see Step 12 workstream).
4. **Next.js 16 unfamiliarity** — `apps/website/AGENTS.md` (a file already present in the uploaded codebase, not user instruction) flags that Next 16 has breaking changes versus older training data and asks to check `node_modules/next/dist/docs/` before writing code. Treated as legitimate in-repo developer guidance; consulted before touching any Route Handler or config in this app.
5. **Two independent Next.js versions in one monorepo** — `apps/dashboard` is Next 15/Tailwind 3, `apps/website` is Next 16/Tailwind 4. This is fine under npm workspaces (independent `node_modules` resolution per package) but means no shared Next config/version assumptions between the two frontends.
6. **Pre-existing lint debt** — 33 errors/57 warnings unrelated to Phase 5. Left as noted below.

## 6. Missing capabilities (confirmed absent, must be built)

- Anonymous session issuance/verification.
- Public webchat router + schemas.
- Guest-side conversation/message persistence path that doesn't require staff auth.
- Rate limiting middleware/dependency (any kind).
- Contact-capture-to-Customer merge logic reusable from the webchat flow (Step 8) — the existing `add_contact` service call is reusable; what's missing is the webchat-specific request schema and duplicate-detection response shaping (never confirming/denying an existing contact per the brief).
- Frontend: env-driven API base URL, session bootstrap/restore, citation rendering, handoff-state UI, sanitized Markdown rendering (assistant responses today are rendered as plain `{msg.text}` — safe already, since it's not `dangerouslySetInnerHTML`, but if Markdown is introduced it must go through a sanitizer).

## 7. Proposed implementation plan

1. **Backend**: new `app/webchat/` module — `models.py` (a `webchat_sessions` table: opaque token hash, customer_id, conversation_id, created_at, last_seen_at, expires_at), `schemas.py`, `service.py`, `router.py` under `/api/v1/webchat/*`, a new Alembic migration. Session resolution via an `HttpOnly`/`Secure`/`SameSite=Lax` cookie holding a high-entropy opaque token (never the raw customer/conversation UUID); the backend looks up the session server-side on every request.
2. **Rate limiting**: a small Redis-backed token-bucket dependency (fail-open — if Redis is down, log and allow, never hard-fail guest chat over an optional dependency) applied to session-creation and message-send endpoints, with limits read from `Settings` (new `WEBCHAT_*` fields), not hard-coded.
3. **Reuse, don't duplicate orchestration**: the webchat message endpoint creates/persists the guest `Message` row (reusing `app.messages` the same way the orchestration router expects), then calls the exact same `orchestrate()` function the staff-facing router calls — same idempotency guarantee, same guardrails, same handoff engine, same citations.
4. **Frontend**: replace `/api/chat/route.ts`'s body with a thin proxy to the FastAPI backend (`NEXT_PUBLIC_API_BASE_URL`) that forwards the session cookie — the browser never talks to FastAPI directly cross-origin for credentials purposes, keeping the existing same-origin `/api/chat` contract `ChatWidget.tsx` already uses, while the real backend does all the work. (Confirmed as the right shape: it also means secrets stay server-side twice over — once in FastAPI's own env, and the Next.js server component never needs any AI credentials either.)
5. **Rebuild `ChatWidget.tsx`'s internals** (state shape, citations, handoff banner, quick actions, retry, mobile full-screen breakpoint, accessibility) while keeping its established visual identity.
6. **Lint debt**: fix the 33 pre-existing errors (unescaped apostrophes, unused imports) as a small, mechanical, non-visual cleanup pass — required for the new `website` CI job to be meaningful, explicitly not a redesign.
7. **CI**: add a `website` job to `.github/workflows/ci.yml` mirroring the existing `dashboard` job (lint, typecheck, build).

## 9. Database migration required?

**Yes, one migration**: a `webchat_sessions` table (opaque-token-hash →
customer_id/conversation_id mapping, expiry). No changes to `customers`,
`conversations`, or `messages` schemas — `channel="webchat"` and anonymous
`Customer` rows (no contacts) are already fully supported by the existing
Phase 2 schema. This will be built, reviewed, and tested against an isolated
schema (per the session's established schema-sandbox test safety controls)
before ever running against the live database, consistent with
`docs/incidents/DATABASE_SAFETY_CONTROLS.md`.
